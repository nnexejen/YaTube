## Введение
Проект YaTube представляет из себя упрощенную социальную сеть, инструмент для начинающих блогеров. 

## Функционал проекта
- Расширенная регистрация пользователей.
- Публикация постов с возможностью добавления изображений.
- Размещение постов в группах.
- Создание групп (только для привилегированных пользователей).
- Комментарии к постам других авторов.
- Подписка на авторов.
- Фильтр постов по группе.

## Используемые технологии
При создании, тестировании и разворачивании приложения использовались следующие технологии:

- `pip-audit` (Статический анализ зависимостей Python)
- `Codecov` (Метрики качества кода)
### Testing & Quality:
- `pytest`, `pytest-django`
- `flake8` (Линтер)
- `Faker`, `mixer` (Генерация тестовых данных)

## CI/CD и безопасность

Проект использует **GitHub Actions** для полной автоматизации проверки кода, сборки образов и контроля безопасности.  

### Пайплайн (Workflow)

Файл конфигурации: `.github/workflows/ci.yml`  
При каждом `push` или `pull request` запускается следующая цепочка задач:
1. **lint** (`flake8`) - проверка стиля кода (PEP8). Исключает миграции и автогенерируемые файлы.     
2. **test** (`pytest`) - запуск автотестов с подключением к PostgreSQL. Включены проверки на уязвимости с использованием инструментов: `Trivy` - сканирование образа на уязвимости, `pip-audit` - сканирование внешних библиотек на наличие уязвимостей.    
3. **docker-build** (`Docker`) - сборка оптимизированного Multi-stage образа. Добавлено управление метаданными: c помощью `docker/metadata-action` автоматически формируются теги образа, применяется система версионирования: короткий SHA-хэш коммита для прослеживаемости и тег `latest` для стабильной версии в ветке `master`.  Добавлено уведомление в ТГ об ошибках при сборке образа.  
Внедрено кэширование слоев в удаленный реестр (`type=registry`). Это сокращает время повторных сборок на 60-80%, так как Docker не пересобирает неизменные этапы (например, установку зависимостей).  
4. **coverage** (`Codecov`) - генерация отчета о покрытии кода тестами.  
5. **deploy-staging** - имитация деплоя (в разработке; добавлены уведомления в ТГ об успешном деплое или падении пайплайна).  

### GitHub Environments & Manual Approval
Для деплоя используется окружение `staging`.  
Настроено ручное подтверждение (Required Reviewers): деплой не начнется автоматически, пока ответственный разработчик не проверит статус сборки и не даст аппрув в интерфейсе GitHub.  

### Локальная проверка пайплайна

Для отладки CI-конфига без отправки кода на GitHub используйте утилиту [`act`](https://github.com/nektos/act) — эмулятор GitHub Actions на базе Docker.

#### Требования
- Установленный Docker
- Утилита `act` ([инструкция по установке](https://nektosact.com/installation/))

#### Запуск тестов локально

Быстрый запуск через скрипт:
```
./run-act.sh
```
Или вручную с параметрами:
```
act -j test \
  -P ubuntu-latest=catthehacker/ubuntu:act-latest \
  -s DB_NAME=yatube \
  -s POSTGRES_PASSWORD=postgres \
  -s SECRET_KEY=test123 \
  -s DB_HOST=localhost \
  -s POSTGRES_USER=postgres \
  -s DB_PORT=5432 \
  -s DEBUG=false
```

### Безопасность и уязвимости  
 
Проект использует Trivy для сканирования Docker-образов. 
В production рекомендуется `exit-code: '1'` для блокировки релиза при наличии критических ошибок.  

**Текущий статус:** 
- Найдено 53 уязвимости (11 CRITICAL, 42 HIGH)
- Основные причины: устаревший Django 2.2.16 и базовый образ

**План исправления:**
- Обновить Django до 3.2+ (требует рефакторинга кода)
- Обновить Pillow до 10.x
- Использовать multi-stage build с регулярным обновлением пакетов

## Переменные окружения

### Локальные переменные 
В директории infra репозитория находятся файлы ```all.env``` и ```web.env```, содержащие следующие переменные окружения:\
```DEBUG``` будет ли включен режим отладки (небезопасно)\
```SECRET_KEY``` секретный ключ Django, при помощи которого будут генерироваться CSRF-токены\
```DB_NAME=postgres``` имя базы данных\
```POSTGRES_USER=postgres``` логин для подключения к базе данных\
```POSTGRES_PASSWORD=postgres``` пароль для подключения к БД (установите свой)\
```DB_HOST=db``` название сервиса (контейнера)\
```DB_PORT=5432``` порт для подключения к БД\
При необходимости измените значения.

### GitHub Actions Secrets

Для корректной работы CI/CD пайплайна необходимо добавить следующие секреты в настройках репозитория (**Settings > Secrets and variables > Actions**):

`DOCKERHUB_USERNAME` логин на Docker Hub  
`DOCKERHUB_TOKEN` Access Token от Docker Hub (не пароль!)  
`TG_BOT_TOKEN` токен вашего Telegram-бота от @BotFather  
`TG_CHAT_ID` ID вашего чата или группы для уведомлений  
`DB_NAME` имя БД  
`POSTGRES_PASSWORD` пароль для подключения к БД  
`SECRET_KEY` секретный ключ Django для генерации токенов  
`CODECOV_TOKEN` токен для интеграции с сервисом Codecov (оценка покрытия кода тестами)

## Запуск приложения
1. Запустите ```minikube``` и ```helm```, создайте namespace для релиза.
2. В отдельной вкладке консоли запустите и не закрывайте
```
minikube tunnel --bind-address 0.0.0.0
```
3. Сгенерируйте сертификаты и поместите их в ```infra/yatube-charts/ssl```
4. Перейдите в директорию ```infra/yatube-charts``` и выполните
```
helm install yatube-release .
```

5. Сайт по умолчанию доступен по имени https://yatube.local (или по имени, которое вы назначили в ```values.yaml (.Values.ingress)```). 
5.1. Если вы используете ```minikube```, имя сайта должно быть записано в файле ```hosts```.
6. Учётная запись суперпользователя создаётся автоматически, параметры входа можно изменить в ```values.yaml (.Values.superuser)```.
